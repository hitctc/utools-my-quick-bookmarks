# 书签标题拼音搜索设计

## 1. 背景

当前书签搜索已经支持：

- uTools 顶部输入框搜索
- 空格分词
- 多关键词 `AND` 匹配
- 标题、域名、目录路径和完整 URL 的原文命中
- 命中片段高亮与“网址命中”提示

但在真实使用里，如果书签标题是中文，用户只能输入中文原文才能命中，这会带来两个明显问题：

- 用户知道标题读音，但不想切换输入法输入中文
- 用户已经习惯在启动器或工具里用拼音首字母快速定位条目

这轮目标不是重做整套搜索系统，而是在保持现有搜索模型不变的前提下，只给“书签标题”补上拼音搜索能力。

## 2. 已确认约束

### 2.1 搜索边界

- 只对“书签标题”增加拼音搜索
- 不对目录路径、根目录文案、域名或完整 URL 增加拼音搜索
- 继续沿用“空格分词 + 多关键词 `AND`”的现有规则
- 不新增模糊排序、权重打分或搜索建议

### 2.2 支持的拼音形式

- 支持无声调全拼，例如 `wangluoanquan`
- 支持拼音首字母，例如 `wlaq`
- 不要求支持带音调拼音输入
- 不要求支持首字母与分隔符的多种变体格式

### 2.3 展示与高亮

- 原文命中时继续沿用现有高亮逻辑
- 纯拼音命中时，先只保证“能搜到”
- 本轮不做“拼音命中时反向高亮中文字符”的复杂映射

### 2.4 技术约束

- 当前仓库没有现成拼音依赖
- 不手写中文到拼音的本地映射表，避免维护成本和错误率
- 继续保持浏览器侧可运行，不引入服务端搜索步骤

## 3. 本轮要实现

- 书签标题支持中文原文、无声调全拼和拼音首字母三种匹配方式
- 现有多关键词 `AND` 搜索规则继续生效
- 拼音能力只作用在标题字段，不外溢到其他搜索字段
- 顶部输入提示文案更新为包含拼音能力的说明
- 为标题拼音匹配补充单测和手动 smoke test

## 4. 本轮明确不做

- 不给目录路径增加拼音搜索
- 不给完整 URL、域名增加拼音搜索
- 不做拼音命中高亮
- 不做搜索排序升级
- 不做搜索历史、联想词或候选提示
- 不为多音字提供手动配置 UI

## 5. 方案选择

本轮采用成熟拼音库生成标题搜索别名，而不是维护自定义字典。

推荐依赖：

- `pinyin-pro`

选择原因：

- 在浏览器侧直接可用，适合当前 `Vue + Vite` 结构
- 同时支持无声调全拼和拼音首字母
- API 足够简单，接入层代码可以保持很薄
- 相比自己维护映射表，更可靠也更容易测试

文档来源：

- Context7 `/zh-lx/pinyin-pro`

## 6. 搜索行为设计

## 6.1 标题搜索别名

对每个书签标题，搜索层额外生成两份规范化别名：

- `fullPinyin`：标题对应的无声调全拼连续串
- `initials`：标题对应的无声调首字母连续串

例如标题为“网络安全”时：

- 原文：`网络安全`
- 全拼：`wangluoanquan`
- 首字母：`wlaq`

英文、数字和符号保留原有原文匹配能力，不额外生成拼音。

## 6.2 Token 匹配规则

每个 token 仍然必须命中同一条书签的任一允许字段，整条书签才算命中。

允许命中的字段变为：

- 标题原文
- 标题全拼别名
- 标题首字母别名
- 完整 URL
- 目录路径
- 域名

其中：

- 标题的拼音能力只补充标题字段的命中面
- 其他字段仍只做原文包含匹配

## 6.3 混合搜索示例

以下输入都应成立：

- `网络`
- `wangluo`
- `wl`
- `wangluo 安全`
- `wl 安全`

以下输入不应因为非标题字段被错误扩展：

- 目录路径是“工作学习”，标题不含相关中文时，`gongzuo` 不应命中
- 域名是 `weixin.qq.com` 时，`weixin` 只作为原文域名匹配，不属于拼音能力

## 7. 高亮与命中反馈

## 7.1 原文命中

如果 token 命中了标题原文、域名、路径或完整 URL 的现有可见字段逻辑：

- 保持当前高亮策略不变

## 7.2 纯拼音命中

如果 token 只命中了标题的全拼或首字母别名，而没有命中标题原文：

- 结果照常展示
- 标题文本保持普通显示
- 不新增额外“拼音命中”角标

这样做的原因是：

- 本轮重点是“搜得到”，不是“拼音字符级高亮”
- 反向把 `wangluo` 映射回“网络”会引入额外复杂度和误判
- 可以避免搜索层和展示层强耦合

## 8. 实现设计

## 8.1 模块边界

搜索能力继续收口在：

- `src/bookmarks/search.js`

不把拼音转换逻辑散到组件层或 `App.vue`。

## 8.2 计划新增的辅助能力

在 `search.js` 内增加一个标题搜索别名辅助层，职责包括：

- 读取标题原文
- 生成无声调全拼
- 生成无声调首字母
- 对结果做轻量缓存

推荐输出结构：

- `normalizedTitle`
- `titleFullPinyin`
- `titleInitials`

## 8.3 缓存策略

拼音转换不应该在每次渲染都重复执行。

因此本轮使用“按标题字符串缓存”的轻量策略：

- 相同标题文本复用同一份拼音结果
- 缓存作用域仅在前端当前会话内
- 不把拼音索引写入本地存储

这样可以降低首页重渲染、筛选和键盘移动时的重复计算成本，同时保持实现简单。

## 8.4 输入提示文案

uTools 顶部输入提示文案同步更新，使用户能感知拼音能力已经上线。

文案方向：

- `搜索书签标题、网址或目录，标题支持全拼和拼音首字母`

如果后续觉得过长，可以再压缩，但本轮优先准确表达能力边界。

## 9. 测试设计

相关单测继续放在：

- `tests/preload/search.test.mjs`

至少补以下场景：

- 中文标题可被全拼命中
- 中文标题可被首字母命中
- 中文标题在混合中文与拼音 token 时仍满足 `AND` 语义
- 纯拼音命中标题时结果可见，但标题高亮仍保持普通文本
- 目录路径不会因为拼音能力被误命中

## 10. 手动验证

本轮 smoke test 至少包括：

1. 输入中文标题原文，确认结果正常命中
2. 输入同一标题的全拼，确认结果命中
3. 输入同一标题的拼音首字母，确认结果命中
4. 输入“中文 + 拼音”混合关键词，确认仍按 `AND` 语义过滤
5. 输入目录路径的拼音，确认不会误命中
6. 用上下键切换结果并继续输入，确认前一轮修复的输入框聚焦行为不回退

## 11. 风险与取舍

- 多音字仍依赖拼音库默认策略，本轮不提供人工纠正入口
- 纯拼音命中不做高亮，短期内用户可用性优先于“命中反馈完整性”
- 新增依赖会略微增加构建体积，但相对当前插件规模可接受

## 12. 交付结论

这轮以“只给标题加拼音搜索别名”为边界，是一次低风险、可验证、对用户搜索效率直接有帮助的小扩展。

它不会改变现有搜索的主规则，也不会把拼音逻辑扩散到 URL、域名和目录路径，适合直接进入实现计划阶段。
