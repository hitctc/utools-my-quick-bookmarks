# 书签缓存秒开与手动刷新设计

## 1. 背景

当前书签插件每次进入首页都会立即同步读取本地 `Chrome Bookmarks` 文件。

现状问题：

- 进入插件时首页会先进入整屏 `loading` 状态
- 已经打开过一次的用户，二次进入仍然要等待文件读取
- 在 uTools 小窗口里，这种白屏闪烁感非常明显
- 当前首页没有直接的“刷新书签”入口，用户只能靠重新读取路径或重进插件

这轮的目标不是改动书签解析能力，而是优化“打开即用”的体验。

## 2. 已确认方案

用户已确认采用：

- 先显示缓存
- 再后台静默刷新
- 右下角增加手动刷新按钮

也就是：

1. 如果本地已有上次成功解析的书签结果，进入插件时直接显示缓存内容
2. 同时后台读取真实书签文件
3. 如果真实文件有变化，再无白屏地更新页面
4. 用户可以通过右下角按钮主动刷新

## 3. 目标

- 避免二次进入插件时出现整屏白屏闪烁
- 让首页在已有缓存时做到秒开
- 保持数据最终仍能和真实书签文件同步
- 增加用户可见、可控的手动刷新入口
- 出错时优先保留可用旧数据，不因为刷新失败而把首页清空

## 4. 不做范围

- 不做缓存过期时间策略
- 不做增量 diff 或部分更新
- 不做多 profile、多浏览器支持扩展
- 不改动现有搜索、置顶、最近打开、打开次数逻辑
- 不新增页面内加载骨架屏系统

## 5. 数据设计

## 5.1 新增缓存对象

新增一份书签读取缓存，保存到 `utools.dbStorage`。

建议字段：

- `filePath`
- `total`
- `items`
- `cachedAt`

其中：

- `items` 直接保存最近一次成功解析出的书签数组
- `cachedAt` 只用于后续展示或排查，本轮不用于过期判断

## 5.2 写入规则

只有在真实文件读取并解析成功后，才写入缓存。

以下情况不覆盖缓存：

- 当前路径不存在
- 文件不可读
- JSON 无效
- 文件可读但没有解析出书签

这样可以避免一次失败把用户上次正常缓存冲掉。

## 5.3 读取规则

插件初始化时：

1. 先读取缓存
2. 如果缓存有效，则立即渲染缓存结果
3. 再发起真实文件静默刷新

缓存有效的判断以结构合法为准，不额外校验文件是否仍存在。

## 6. 首页打开流程

## 6.1 首次进入且无缓存

行为保持接近当前：

- 显示加载态
- 读取真实书签文件
- 成功后渲染并写缓存
- 失败则展示错误态

## 6.2 已有缓存时

进入插件后的顺序改为：

1. 立即显示缓存结果
2. 不进入整屏白屏 `loading`
3. 后台静默读取真实文件
4. 如果结果变化，则更新页面和缓存
5. 如果结果未变化，则保持当前界面不动
6. 如果读取失败，则保留缓存内容，只记录错误提示

## 6.3 视图状态区分

首页需要把“首次阻塞加载”和“静默刷新中”区分开：

- `blockingLoading`
  仅用于首次无缓存时的整屏加载态
- `refreshing`
  用于已有缓存时的后台刷新和手动刷新

这两个状态不能继续共用同一个 `loading` 语义，否则会再次触发白屏切换。

## 7. 刷新策略

## 7.1 自动静默刷新

以下时机触发自动静默刷新：

- 插件初始化完成后，且已经成功显示缓存
- 从设置页返回首页后，如果路径或书签源可能有变化

静默刷新期间：

- 首页保持当前内容
- 不切整屏加载态
- 右下角可显示简短状态，例如 `刷新中`

## 7.2 手动刷新

首页右下角新增独立的 `刷新书签` icon 按钮。

交互规则：

- 点击后强制读取当前路径对应的真实书签文件
- 成功则更新页面和缓存
- 失败则保留当前内容，并提示刷新失败
- 刷新进行中按钮进入禁用态，避免连续触发

## 7.3 是否更新页面的判断

真实读取成功后，需要判断是否真的有变化，再决定是否替换页面数据。

本轮建议采用“稳定签名对比”或“序列化结果对比”的简单方案：

- 对 `filePath + total + items` 生成稳定字符串
- 新旧一致则不更新页面
- 新旧不同则替换页面数据并回写缓存

本轮不引入复杂 diff。

## 8. 交互与视觉

## 8.1 右下角浮层

当前右下角浮层已有状态条和设置按钮。

本轮调整为：

- 状态条
- 刷新按钮
- 设置按钮

三个元素独立排列，保持 Nothing 风格一致性。

## 8.2 刷新状态文案

状态条建议支持以下短文案：

- 默认：`跟随 / 浅色`、`上下键选`
- 静默刷新时：增加 `刷新中`
- 刷新失败时：短暂显示 `刷新失败`

不使用整屏弹层，不使用抢占式提示。

## 8.3 首页错误策略

当页面已经有缓存数据时：

- 后台刷新失败不再把首页切换成整屏错误态
- 仅保留当前列表，并用轻量提示反映刷新失败

只有“首次进入且无缓存”同时读取失败时，才显示当前整屏错误态。

## 9. 代码改动范围

预计涉及：

- `public/preload/localState.cjs`
  新增缓存对象归一化逻辑
- `public/preload/services.js`
  新增读取 / 保存书签缓存能力
- `src/App.vue`
  调整初始化流程、阻塞加载与静默刷新状态、缓存优先展示逻辑
- `src/bookmarks/HomeView.vue`
  接收刷新状态、刷新按钮事件
- `src/main.css`
  增加刷新按钮和状态表现
- `tests/preload/localState.test.mjs`
  覆盖缓存对象归一化

如果把缓存归一化单独拆成更清晰模块，也可以新增专用测试文件，但本轮不强制重构。

## 10. 验证

至少需要覆盖：

1. 无缓存首次进入时，仍能正常读取和展示
2. 有缓存再次进入时，不出现整屏白屏
3. 自动静默刷新成功后，页面和缓存同步更新
4. 自动静默刷新失败时，旧内容仍保留
5. 手动刷新成功时，页面立即更新
6. 手动刷新失败时，不清空当前页面
7. 右下角刷新按钮在刷新中禁用

工程验证顺序：

- 最相关单测
- `npm run build`
- uTools 手动 smoke test

## 11. 风险与约束

- 缓存体积会比当前本地状态大，需要接受 `items` 存入 `dbStorage`
- 如果缓存结构变更，必须做归一化，避免旧缓存污染首页
- 静默刷新不能影响当前高亮索引、搜索输入和已展示结果的稳定性
- 如果用户刚改了路径，需要以当前最新路径为刷新目标，不能误读旧路径缓存
